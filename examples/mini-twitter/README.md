# mini-twitter

A small social app written end-to-end in mar: passwordless email
sign-in, handle-based profiles, follow graph, and a personalized
timeline. Around 600 LoC.

The goal is to show what a real Mar app looks like when you put
together everything the framework offers. Read the modules
top-to-bottom in this order:

1. **`mar.json`** — production-ready manifest:
   - `auth.sessionSecret = env:SESSION_SECRET` (auto-generated by
     `mar fly provision`)
   - `mail.*` — SMTP via Resend (the From: address you set here is
     the single source of truth for every framework-sent email)
   - `admins` — one operator email so the admin panel at
     `/_mar/admin` is reachable in prod
   - `adminPanel.recentRequestsSize = 500` — bigger than default so
     a few minutes of traffic stays visible
   - `rateLimit` — 1200 req/min, burst 60 (generous; tighten if
     you see abuse)
   - `server.maxBodyBytes = 65536` — 64 KiB; this app's largest
     payload is a 280-char tweet, so we don't need the 1 MiB default
   - `database.autoBackup` — explicit 6 h interval, 28 retention

2. **`Shared.mar`** — domain types + service contracts. Reading this
   alone tells you the API surface of the app.

3. **`Backend/Users.mar`**, **`Tweets.mar`**, **`Follows.mar`** —
   one file per entity, each just the `Entity.define` call. Schema
   lives close to the domain it models, not in a giant central
   file.

4. **`Backend/Services.mar`** — every business handler + the
   `services` manifest at the bottom. The manifest reads
   top-to-bottom as the security policy of the app:

   ```
   me            : any authed user
   loadTimeline  : any authed user → personalized timeline
   composeTweet  : any authed user → posts as themselves
   deleteTweet   : owner only       (Auth.authorize)
   loadProfile   : any authed user
   follow        : any authed user
   unfollow      : any authed user
   setupProfile  : any authed user
   ```

5. **`Frontend/Routes.mar`** — all URL patterns in one place. The
   typed `Path` values mean refactoring a URL (`{handle:String}` →
   `{userId:Int}`) breaks every call-site at compile time.

6. **`Frontend/SignIn.mar`** — passwordless flow: ask for email,
   send a 6-digit code via SMTP, verify. State machine with
   explicit Submitting/Error states so the UX doesn't flash
   between two screens on failure.

7. **`Frontend/SetupProfile.mar`** — first-time setup: pick handle
   + display name. Reached via `Nav.replaceTo` from Home when the
   logged-in user's handle is empty.

8. **`Frontend/Home.mar`** — the timeline + inline compose box.
   Demonstrates stale-while-revalidate (the list stays visible
   during the re-fetch after Post / Delete, no "Loading…" flash).

9. **`Frontend/Profile.mar`** — public profile by handle. Uses
   `Page.dynamicProtected` because the path has a `{handle}`
   segment; init/update/view all receive the typed params record.

10. **`Main.mar`** — the wiring point: `Auth.config` declares how
    users are identified, what the sign-in email looks like, and
    where to send signed-out visitors; `App.fullstack` mounts the
    services and pages.

## Running locally

```sh
mar dev examples/mini-twitter/
```

Open <http://localhost:3000>. The sign-in code prints to the
terminal (no SMTP needed in dev). The admin panel is at
<http://localhost:3000/_mar/admin>.

## Deploying to Fly

```sh
mar admin add YOUR_EMAIL          # so you have an operator login
cd examples/mini-twitter
mar fly init                      # scaffolds deploy/fly/
mar fly provision                 # creates app + auto-generates SESSION_SECRET,
                                  # prompts for SMTP_PASSWORD (your Resend API key)
mar fly deploy                    # builds + ships
```

The Resend account whose API key you provide must have the domain
in `mar.json["mail"]["from"]` verified (DKIM + SPF + Return-Path);
without verification every send is rejected with `550 ... not
authorized`.

## What this example doesn't have

Kept out on purpose — they'd grow the example past "easy to read
top-to-bottom" without adding language coverage:

- Reply / quote tweet flows (one more entity + threading)
- Tweet edit (just `Repo.update` on top of the delete decorator
  pattern — same shape)
- Followers / Following list pages (one more page each)
- Search (needs a full-text index or external service)
- Notifications (needs an event bus)
- Media uploads (needs streaming routes, beyond the JSON-only
  shape this example uses)
